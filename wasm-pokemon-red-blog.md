# how i turned Pokemon Red into a WebAssembly web app without rebuilding the game

so, quick about us. i got here the way a lot of good technical work starts: with a slightly unhinged idea and a repo that already did something real. this codebase was not a fake demo. it already built an actual Game Boy Color ROM with `rgbds`, and that changed the whole strategy immediately. instead of asking, "how do i remake Pokemon Red for the browser," i asked the better question: "how do i run the real thing in the browser with the least amount of distortion?"

that framing matters a lot. once i looked at the repo, it was obvious the right move was not an HTML/CSS remake and not some giant JS rewrite. the ROM already existed. the assets already existed. the game logic already existed. so the play was to keep the ROM build authoritative, pick a WebAssembly-capable emulator runtime, wrap it in a static web shell, and make the whole thing deploy cleanly to GitHub Pages.

that is also exactly the kind of problem Claude or Codex can crush if you prompt them like an operator instead of like a search box. the useful prompt is not "make this web." the useful prompt is more like: inspect this repo, find the real build path, research the browser runtime options, choose one with save support, implement it, debug the toolchain, run it in a browser, and do not stop until you have visual proof. that level of specificity is what turns an AI assistant into an actual force multiplier.

in this session, that meant a few concrete things. first, we mapped the build graph and confirmed this was a real ROM pipeline, not a web-first app pretending to be one. then we researched emulator options and landed on `binjgb` because it already had an Emscripten path, browser embedding patterns, and the save APIs we needed. after that, we wired a static player around the ROM, patched the staged emulator build for modern CMake and Emscripten behavior, exported the right runtime symbols, and verified the whole thing with Agent Browser until the canvas was actually rendering frames. not "looks fine in theory." actual pixels. actual game boot. that part is huge.

the reason this technique matters is pretty simple. if you try to remake Pokemon Red in HTML and CSS, there are repos out there for that vibe, but you are signing up to hard-code a wild amount of behavior. menus, battle logic, collision, save flow, animation timing, rendering quirks, audio hooks, edge cases from a twenty-plus-year-old game engine, all of it. that can be fun as an experiment, but it is a completely different project. you are rebuilding a game, not shipping a browser runtime for one.

with WebAssembly, the posture changes. you can take a ROM that is only a few megabytes, pair it with a compact emulator runtime, and run the actual game loop in the browser. now the browser layer is mostly about orchestration: loading binaries, mapping input, presenting video, unlocking audio, persisting SRAM, handling save-state import/export, and packaging the result as a static site. that is way cleaner. and, assuming you have the rights to the ROM or source you are shipping, the legal surface is also cleaner than hand-porting Nintendo-adjacent assets and behavior into a bespoke clone.

WebAssembly is the unlock here because it gives you a bunch of wins at once. it is portable, so the same compiled runtime can boot in every major browser without asking users to install anything weird. it is fast enough for emulator workloads that would feel messy in pure DOM code. it plays nicely with typed arrays and browser APIs, which makes framebuffer and audio handoff pretty straightforward. and it keeps the original execution model intact. you are not approximating the game. you are running the game.

there is also a product lesson in here. the best use of Claude or Codex on hard problems is not just code generation. it is decomposition plus verification. ask for the plan. ask for the runtime tradeoff. ask it to inspect the repo instead of guessing. ask it to browse for the upstream docs. ask it to compile. ask it to run the page in a browser and take a screenshot. that sequence is way more valuable than one giant "build me x" prompt.

for this project, the end state was honestly kind of sick in a very practical way: a ROM-driven web build, a static `dist/web` artifact, root `index.html` support for GitHub Pages, IndexedDB-backed persistence, and a deployment workflow that can publish the site from `main`. no giant frontend rewrite. no pretending HTML/CSS is a game engine. just a thin, technically honest shell around the original binary.

that is the bigger takeaway for me. when the source artifact is already real, do not throw it away just to make the architecture feel modern. use WebAssembly, keep the core intact, and let the browser do what it is actually good at. that is not just cleaner engineering. it is faster, more robust, and, low-key, the difference between a cute demo and a shippable system.
